Point de départ

Comment apprendre avec ce guide

Ce manuel n'est pas une liste de blocs à copier. Le but est de t'aider à passer de « je reconnais ce code » à « je sais le reconstruire et expliquer pourquoi il fonctionne ».

Une méthode simple

  1. Lis une petite partie sans ouvrir GameMaker.
  2. Prédis le résultat du code.
  3. Teste-le dans un projet minuscule.
  4. Change une valeur et observe ce qui se passe.
  5. Refais l'exercice sans regarder la solution.
  6. Consulte le manuel officiel pour vérifier les détails.
Une erreur comprise est utile. Un code copié qui fonctionne sans que tu saches pourquoi sera beaucoup plus difficile à modifier plus tard.

Quand tu bloques

Réduis le problème. Affiche une variable, désactive temporairement une animation, retire un obstacle ou teste une seule direction. Le debug consiste souvent à enlever des inconnues jusqu'à ce qu'il ne reste qu'une cause possible.

Leçon 1

Les briques de GameMaker

ÉlémentRôleExemple
SpriteUne image ou une animation. Il ne contient pas la logique du jeu.spr_player_idle
ObjetUn modèle qui contient des événements et des variables.obj_player
InstanceUne copie vivante d'un objet, placée ou créée dans une room.Le joueur présent dans le niveau
RoomUn espace qui contient des calques, des instances, des tilemaps et une caméra.rm_level
ÉvénementUn moment où GameMaker exécute du code.Create, Step, Draw, Collision

Create et Step

L'événement Create s'exécute lorsque l'instance apparaît. C'est un bon endroit pour initialiser ses variables.

// Create de obj_player
move_speed = 2;
hp = 3;
facing = "down";

Le Step s'exécute à chaque cycle du jeu. On y lit souvent les commandes et on met à jour la logique.

Objet et instance ne sont pas synonymes. Tu peux créer un seul objet obj_enemy et placer vingt instances différentes de cet objet dans une room.
Documentation officielle : Objects and Instances.
Leçon 2

Premiers repères en GML

Une variable conserve une valeur

hp = 3;
name = "Player";
is_dead = false;

Les noms décrivent le rôle des valeurs. Une variable comme hp peut changer au fil de la partie. Une variable comme max_hp peut servir de limite.

Variable d'instance ou variable locale

// Variable d'instance, encore disponible au prochain Step
facing = "right";

// Variable locale, limitée au bloc de code actuel
var _horizontal = 1;

Le préfixe souligné n'est pas imposé par GameMaker. C'est une convention pratique pour reconnaître rapidement une variable locale.

Une condition choisit un chemin

if (hp <= 0)
{
    is_dead = true;
}

Le point-virgule

Le point-virgule marque la fin d'une instruction. GameMaker accepte parfois son absence, mais l'utiliser régulièrement rend le code plus lisible et évite des surprises.

Petit exercice. Crée trois variables dans Create, affiche-les avec show_debug_message(), puis modifie une valeur dans Step quand tu appuies sur une touche.
Documentation officielle : Variables and Variable Scope.
Leçon 3

Créer un déplacement top-down

Lire ZQSD

var _horizontal = keyboard_check(ord("D")) - keyboard_check(ord("Q"));
var _vertical = keyboard_check(ord("S")) - keyboard_check(ord("Z"));

Chaque appel retourne zéro ou un. La soustraction donne donc une valeur comprise entre moins un et un.

TouchesValeur horizontale
Aucune0
D1
Q-1
Q et D0

Le piège de la diagonale

Ajouter directement la valeur horizontale et la valeur verticale fait parcourir davantage de distance en diagonale. On peut transformer le vecteur d'entrée en angle, puis utiliser une longueur constante.

if (_horizontal != 0 or _vertical != 0)
{
    var _direction = point_direction(0, 0, _horizontal, _vertical);

    x += lengthdir_x(move_speed, _direction);
    y += lengthdir_y(move_speed, _direction);
}

Avec ce système, move_speed représente la distance totale parcourue, quelle que soit la direction.

Observer les angles

show_debug_message(_direction);

Place ce message dans le bloc où _direction est créée, car cette variable locale n'existe pas en dehors de ce bloc. Dans GameMaker, la droite correspond à 0 degrés, le haut à 90, la gauche à 180 et le bas à 270.

À tester. Affiche la direction dans la console et vérifie les quatre directions. Essaie ensuite deux touches à la fois et note les valeurs obtenues.
Documentation officielle : Keyboard Input et point_direction().
Leçon 4

Mémoriser la direction et choisir une animation

Changer le sprite pendant le mouvement ne suffit pas à savoir quelle direction afficher quand le personnage s'arrête. Il faut conserver cette information dans une variable d'instance.

// Create
facing = "down";
// Step
if (_horizontal > 0)
{
    sprite_index = spr_player_walk_right;
    facing = "right";
}
else if (_horizontal < 0)
{
    sprite_index = spr_player_walk_left;
    facing = "left";
}

À l'arrêt, un switch peut choisir le sprite idle à partir de la dernière valeur mémorisée.

switch (facing)
{
    case "right":
        sprite_index = spr_player_idle_right;
    break;

    case "left":
        sprite_index = spr_player_idle_left;
    break;

    case "up":
        sprite_index = spr_player_idle_up;
    break;

    default:
        sprite_index = spr_player_idle_down;
    break;
}
Erreur fréquente. Le sprite de marche change correctement, mais facing reste toujours égal à "down". Dans ce cas, le personnage revient vers le bas dès qu'il s'arrête.

Pour diagnostiquer ce problème, affiche la variable au moment de l'arrêt :

show_debug_message("facing = " + string(facing));
Documentation officielle : switch et sprite_index.
Leçon 5

Séparer le code avec une fonction

Quand le Step commence à gérer la mort, les commandes, le mouvement, les animations et les collisions, il devient difficile à relire. Une fonction peut isoler une responsabilité précise.

function update_animation(_horizontal, _vertical)
{
    if (_horizontal > 0)
    {
        sprite_index = spr_player_walk_right;
        facing = "right";
    }
    else if (_horizontal == 0 and _vertical == 0)
    {
        // Choisir ici le sprite idle avec facing
    }
}

Le Step calcule les entrées, puis les transmet :

update_animation(_horizontal, _vertical);

Pourquoi passer des paramètres

Une variable déclarée avec var dans le Step est locale à ce bloc. La fonction ne la reçoit pas par magie. Les paramètres rendent les dépendances visibles.

Erreur fréquente. Un else placé après l'accolade qui ferme la fonction n'a plus de if auquel se rattacher. Quand le compilateur indique une erreur près d'une accolade, vérifie aussi les lignes précédentes.
Exercice. Commence avec une fonction qui affiche seulement « marche » ou « idle ». Quand ce test fonctionne, déplace progressivement le choix des sprites dans la fonction.
Leçon 6

Caméra, sous-pixels et pixel art

Un déplacement diagonal normalisé produit souvent des coordonnées décimales. C'est normal pour la logique du jeu, mais cela peut créer un tremblement visuel avec une caméra qui suit le personnage.

x = 290.99;
y = 95.16;

Diagnostiquer avant de corriger

  1. Fige l'animation. Si le tremblement reste présent, l'animation n'est probablement pas la cause.
  2. Désactive le suivi de caméra. Si le tremblement disparaît, la caméra est impliquée.
  3. Affiche les coordonnées pour vérifier si le mouvement produit des fractions de pixel.

Garder une position logique précise

Il ne faut pas forcément arrondir directement x et y. Cela changerait le déplacement. On peut arrondir uniquement le dessin.

// Draw de obj_player
draw_sprite(
    sprite_index,
    image_index,
    round(x),
    round(y)
);

La position logique reste décimale pour les mouvements et les collisions. La position visuelle reste alignée sur la grille de pixels.

Une caméra contrôlée dans End Step

var _camera = view_get_camera(0);
var _view_width = camera_get_view_width(_camera);
var _view_height = camera_get_view_height(_camera);

var _camera_x = round(obj_player.x - _view_width / 2);
var _camera_y = round(obj_player.y - _view_height / 2);

camera_set_view_pos(_camera, _camera_x, _camera_y);

Une version complète doit aussi empêcher la caméra de sortir de la room avec clamp(). Le End Step est utile car il s'exécute après le déplacement normal du joueur.

Pour un rendu net, une vue logique de 320 par 180 peut être agrandie en 1280 par 720. Le facteur est exactement égal à quatre.
Documentation officielle : camera_set_view_pos() et draw_sprite().
Leçon 7

Premières collisions avec objets et tilemaps

place_meeting() pose une question : si l'instance était à cette position, rencontrerait-elle un obstacle ? La fonction retourne vrai ou faux, mais elle ne déplace rien.

if (!place_meeting(x + _dx, y, obj_solid))
{
    x += _dx;
}

move_and_collide() tente de déplacer l'instance et ajuste son mouvement lorsqu'elle rencontre l'obstacle donné.

move_and_collide(_dx, _dy, obj_solid);

Collision avec une tilemap

Une couche de tiles n'est pas un objet. Il faut d'abord récupérer son identifiant.

// Create
collision_tilemap = layer_tilemap_get_id("Tiles_Collision");
// Step
move_and_collide(_dx, _dy, collision_tilemap);

Le nom de couche doit correspondre exactement au nom utilisé dans la room. Le tileset doit également posséder un sprite source et un masque utilisable pour les collisions.

Stocker seulement la chaîne "Tiles_Collision" ne donne pas un Tile Map ID. La chaîne sert à rechercher la couche avec layer_tilemap_get_id().
Protocole de test. Approche un bord horizontalement, verticalement puis en diagonale. Vérifie ensuite les coins et les vitesses plus élevées.
Documentation officielle : place_meeting(), move_and_collide() et Collisions.
Leçon 8

Afficher les PV dans Draw GUI

Le Draw GUI utilise une surface d'interface qui ne se déplace pas avec la caméra. C'est l'endroit naturel pour les PV, le score et les menus.

if (instance_exists(obj_player))
{
    for (var i = 0; i < obj_player.hp; i++)
    {
        draw_sprite(spr_health, 0, 8 + i * 18, 8);
    }
}

La boucle commence à zéro. Tant que i reste inférieur aux PV, une nouvelle icône est dessinée. Le terme i * 18 décale chaque icône.

Pourquoi vérifier l'existence du joueur

Une future room de titre peut ne contenir aucun joueur. Sans vérification, lire obj_player.hp pourrait provoquer une erreur.

État du dessin

draw_set_alpha(0.75);
draw_set_colour(c_black);
draw_rectangle(0, 0, 320, 180, false);

draw_set_alpha(1);
draw_set_colour(c_white);

Changer l'alpha ou la couleur affecte les dessins suivants. Il est donc utile de remettre un état connu avant de dessiner le reste de l'interface.

Documentation officielle : Drawing and the GUI Layer et draw_sprite().
Leçon 9

Dégâts, invincibilité et Game Over

Après un dégât, une courte invincibilité évite de retirer tous les PV pendant plusieurs frames de collision.

// Create du joueur
hp = 3;
max_hp = 3;
invincibility = 0;
is_dead = false;
// Lors d'un dégât
if (other.invincibility <= 0)
{
    other.hp = max(0, other.hp - 1);
    other.invincibility = 60;

    if (other.hp <= 0)
    {
        other.is_dead = true;
    }
}

À 60 steps par seconde, une valeur de 60 représente environ une seconde. Cela reste un réglage de game design, pas une règle absolue.

Arrêter le Step après la mort

if (is_dead)
{
    visible = true;

    if (keyboard_check_pressed(ord("R")))
    {
        room_restart();
    }

    exit;
}

exit arrête l'événement actuel. Il ne ferme pas le jeu. À la frame suivante, le Step recommence depuis sa première ligne.

Erreur fréquente avec max. max(0, hp, -1) conserve souvent les PV actuels, car ils sont la plus grande valeur. Pour retirer un PV, la soustraction doit se trouver dans l'argument : max(0, hp - 1).
À vérifier. Teste la suite 3, 2, 1, 0. Vérifie que le joueur ne bouge plus, que le texte apparaît et que R remet correctement les variables à leur état initial.
Leçon 10

Versionner un projet GameMaker avec Git

GameMaker fournit une interface de contrôle de source, mais le moteur utilise le véritable Git installé sur la machine.

ActionRôle
Save AllSauvegarder les ressources GameMaker sur le disque.
StageChoisir les changements du prochain commit.
CommitCréer un point d'historique local.
PushEnvoyer les commits locaux vers le dépôt distant.
PullRécupérer les changements du dépôt distant.

Un commit raconte une étape

Add player movement
Add health display
Fix pixel art camera jitter

Un bon commit regroupe une modification cohérente et testée. Il n'est pas nécessaire d'attendre la fin de la journée.

Précaution avec les outils externes

Avant un pull, un changement de branche ou une modification extérieure des fichiers, sauvegarde et ferme le projet dans GameMaker. Tu peux laisser l'IDE ouvert sur sa page d'accueil.

Documentation officielle : Setting Up And Using Source Control.
